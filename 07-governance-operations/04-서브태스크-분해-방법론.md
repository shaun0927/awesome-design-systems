# 서브태스크 분해 방법론

## 📌 핵심 개념

### 1. 문제 정의
**상황**:
- 디자인 시스템 개발 시 상위 단계(Propose, Design, Code, Doc)는 정의되어 있음
- 하지만 팀원들마다 각 단계를 "완료"하는 기준이 달랐음
- 세부 서브스텝이 일관되게 완료되지 않아 결과물의 품질이 불안정함

**해결책**:
- 각 단계를 **Phase(단계)**로 분해
- Phase별로 **Subtask(서브태스크)** 체크리스트 제공
- 예측 가능하고 견고한(robust) 결과물 생산

### 2. 핵심 구조: Step → Phase → Subtask

**6가지 공통 Phase**:
| Phase | 설명 | 소요 시간 |
|-------|------|-----------|
| **START** | 파일 설정(템플릿), 재사용 가능한 것들 감사 | 짧음 |
| **COMPOSE** | 실제 "작업"을 수행하는 단계 (가장 핵심) | 가장 길음 |
| **TEST** | 품질 기준에 따라 검증 | 중간~길음 |
| **PRESENT** | 커뮤니티와 대면 논의 | 짧음~중간 |
| **REVIEW** | 동료 검토, 피드백 및 수정 | 중간 |
| **DELIVER** | 최종 편집, 다음 단계 준비 | 짧음 |

### 3. 4가지 주요 Step
```
Propose → Design → Code → Doc → Release
```

## 🎯 실무 노하우

### 1. 담당자 배정 (Assignment)
**원칙**: Step당 "Main Owner" 배정 + 다른 사람 통합

| 상황 | 담당 범위 |
|------|----------|
| **시스템 팀 개발자** | 모든 단계 직접 수행 (REVIEW만 타인) |
| **외부 기여 개발자** | START, COMPOSE는 본인, TEST/PRESENT는 파트너, REVIEW/DELIVER는 시스템 팀 |

### 2. 도구 통합 (Tool Integration)
**최선: 도구에 Subtask 내장**
- 예: Storybook + Chromatic.com
- 충분한 리뷰어 자동 확인, 미해결 코멘트 확인, 워크플로우 완료 전까지 머지 차단

**차선: Jira/Asana에 체크리스트**
```markdown
# Propose Subtasks
- Discover requirements / assignee:"@current"
- Obtain input on scope / assignee:"@current"
- Document scope and plan / assignee:"@current"
- Present in system team critique / assignee:"@current"
```

### 3. 압도되지 않도록 관리
**해결책**:
1. 처음에는 Phase 1~2개만 추가 (예: START + COMPOSE)
2. 나중에 나머지 Subtask 추가 (TEST, PRESENT, REVIEW, DELIVER)
3. 작은 작업(결함 수정, 기능 개선)에는 Subtask 축소

### 4. 커뮤니티 참여 조율
| 상황 | Phase 배치 |
|------|-----------|
| API 리뷰가 시작을 막는 경우 | **START**로 이동 (COMPOSE 전) |
| 가벼운 피드백 예상 | **TEST, PRESENT, REVIEW**로 미루기 |
| 필요하지 않음 | 완전히 제외 |

## 📋 워크플로우 분해 템플릿

### Propose Step
#### START
- [ ] 제안 템플릿 작성 (What? Why? For Who? By When?)
- [ ] 경험 전반에 걸친 감사(audit)

#### COMPOSE
- [ ] 기능 방향성 및 범위 작성 (Must/Should/Could/Won't)
- [ ] 의존성(Dependencies) 명시

#### REVIEW
- [ ] 시스템 팀 멤버 리뷰
- [ ] 제품 오너 리뷰

#### PRESENT
- [ ] 시스템 팀에 발표
- [ ] 디자인/개발자 커뮤니티에 발표

#### DELIVER
- [ ] 최종 범위 확정
- [ ] 다음 단계 계획 (Jira 티켓 생성)

### Design Step
#### START
- [ ] 심볼 로딩된 템플릿에서 시작
- [ ] 다른 소스에서 아트 통합

#### COMPOSE
- [ ] 기능 디자인 (variations, states, responsiveness)
- [ ] 디자인 토큰과 완전히 통합
- [ ] 주석(annotation) 추가

#### TEST
- [ ] 접근성 테스트 (필수)
- [ ] 사용성 테스트 (선택)

#### PRESENT/REVIEW/DELIVER
- [ ] 시스템 팀 및 커뮤니티 발표
- [ ] 개발자, 접근성 전문가, 콘텐츠 전략 리뷰
- [ ] 아트워크 및 주석 완료

### Code Step
#### START
- [ ] 코드 브랜치 생성
- [ ] 파일 생성
- [ ] 재사용 가능한 코드 소싱

#### COMPOSE
- [ ] API 초안 작성 및 리뷰
- [ ] HTML, CSS, JavaScript 코딩
- [ ] 시각 테스트 생성
- [ ] API 문서화

#### TEST
- [ ] 기능 테스트
- [ ] 브라우저 및 디바이스 테스트
- [ ] 단위 테스트
- [ ] 시각적 회귀 테스트
- [ ] 접근성 테스트 (자동+수동)

#### PRESENT/REVIEW/DELIVER
- [ ] 디자이너 및 시스템 팀 발표
- [ ] 개발자 리뷰 (코드 검사, 테스트 실행)
- [ ] Pull Request 머지, 체인지로그 작성

### Doc Step
#### START/COMPOSE
- [ ] 템플릿에서 문서 파일 설정
- [ ] 컴포넌트 소개 작성
- [ ] 실제 콘텐츠를 사용한 예제 작성
- [ ] 디자인/접근성/콘텐츠 가이드라인 작성

#### REVIEW/DELIVER
- [ ] 시스템 개발자/디자이너/접근성 전문가 리뷰
- [ ] Markdown 마이그레이션, 발행 준비

## 💡 주요 인사이트

1. **체크리스트가 주는 가치**: 견고성, 품질, 신뢰 획득
2. **균형의 중요성**: 너무 많지도 적지도 않게, 점진적 성장
3. **Phase별 시간 분배**: COMPOSE (40%), TEST/REVIEW (각 15%), 나머지 (각 10%)
4. **성공 지표**: 모든 팀원이 "완료" 정의를 동일하게 이해, 외부 기여자가 가이드 없이 작업 가능, 품질 문제 재작업 감소
