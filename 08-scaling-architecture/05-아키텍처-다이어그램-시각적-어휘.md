# 디자인 시스템 아키텍처 다이어그램 (Nathan Curtis, 2019)

## 📌 핵심 개념

### 문제 정의
대부분의 기업은 "하나의 디자인 시스템이 모든 제품을 지원"하는 단순한 구조를 넘어선 **복잡한 생태계**를 가지고 있다. 이러한 복잡한 시스템의 폭과 다양한 고객에게 제공하는 기능을 빠르게 보여주고 가르칠 수 있는 **시각적 어휘가 부족**하다.

### 솔루션: 시각적 심볼 체계
**브랜드(Brands), 시스템(Systems), 제품(Products)을 연결하는 시각적 어휘**

## 🎯 시각적 심볼 체계

### 1. 시스템 = 다이아몬드 (◆)
```
◆ = 디자인 시스템 기능 세트
- 정의된 비주얼 스타일, UI 컴포넌트, 디자인/코드 자산
```

### 2. 시스템 도구 표시
```
D (빨강, 왼쪽 반) = 디자인 자산 (Sketch, Figma)
C (파랑, 오른쪽 반) = 코드 라이브러리 (React, HTML)
D + C (둘 다) = 양쪽 모두 제공
```

**실제 사례**:
- IBM Carbon, Shopify Polaris → D + C 모두 제공
- Financial Times Origami → C만 제공
- Google Material (과거) → D만 제공

### 3. 문서화 배경 (노란색)
```
왼쪽 노란색 = 디자인 가이드라인
오른쪽 노란색 = 코드 레퍼런스
전체 노란색 = 양쪽 모두 제공
```

### 4. 제품 = 원형 (●)
```
● = 디지털 경험 또는 그 일부
예: Home Page, Search, Cart, Checkout
```

### 5. 제품 도입 상태 (색상)
```
검정(Black) = 시스템 코드 사용하여 채택
회색(Gray) = 시스템 디자인을 맞춤 코드에 임베드
흰색(White) = 시스템 미채택
```

### 6. 브랜드 = 정사각형 (■)
```
■ = 브랜드 아이덴티티
- 색상, 타이포그래피, 일러스트레이션
- 모든 디자인 시스템은 브랜드로부터 상속받음
```

### 7. 조직 경계 (보라색 점선)
```
보라색 점선 컬럼 = 조직 경계
- 도메인, 부족, 스쿼드, 사업부 등
```

## 📊 아키텍처 다이어그램 유형

### 1. 단순형 (Simple)
```
[Brand]
   |
[System] (D+C)
   |
[Products] (N개)
```
**적용**: 스타트업, 단일 제품군

### 2. 중개형 (Intermediary)
```
[System] (D+C, HTML/CSS)
   |
┌──┼──┐
│  │  │
React Vue Direct
trans trans adopt
```
**문제**: 번역 레이어 중복
**해결**: Web Components로 전환

### 3. 다중 시스템 - 수평형
```
      [Brand]
         |
    ┌────┼────┐
[System1][System2][System3]
  .com   Sales  Servicing
```
**기준**: 고객 여정 단계
**적용**: 은행, 보험사

### 4. 다중 시스템 - 수직형
```
      [Brand]
         |
    ┌────┼────┐
[Consumer][Employer][Institutional]
```
**기준**: 고객 유형
**전략**: 자율성 유지, 실무 공유

### 5. 다중 브랜드형
```
[Core Brand]
   |
[Division Brand]
   |
[Design System]
```
**예시**: Marriott (Courtyard, Renaissance, JW)

### 6. 확장/인수형
```
[Core Brand]
   |
[Division Brand]
   |
[Design System]
   |
┌───┴───┬───┬───┐
Code  A Ext B Ext
Lib   Sketch Code
```
**특징**: 인수합병 대비 구조

## 💡 실무 적용

### 다이어그램 작성 전 질문
- [ ] 우리 조직에는 몇 개의 디자인 시스템이 있는가?
- [ ] 각 시스템은 어떤 제품을 지원하는가?
- [ ] 브랜드 아이덴티티는 어디서 관리되는가?
- [ ] 디자인 자산(D)과 코드(C) 중 무엇을 제공하는가?
- [ ] 제품별 시스템 도입 상태는?
- [ ] 조직 경계는 어떻게 나뉘는가?

### 다이어그램 활용
- **경영진 보고**: 시스템 투자 정당화
- **팀 온보딩**: 새 멤버에게 생태계 설명
- **로드맵 계획**: 확장 우선순위 결정
- **중복 식별**: 불필요한 시스템 발견
- **인수합병**: 통합 시나리오 시뮬레이션

### 유지보수 주기
- **분기별**: 제품 도입 상태 업데이트
- **반기별**: 시스템 기능 검토
- **연간**: 조직 구조 변경 반영

## 🎓 3가지 황금률

1. **단순성 추구**: 복잡한 생태계를 한눈에
2. **상태 명확화**: 제품 도입 상태를 색상으로 즉시 파악
3. **조직 반영**: 실제 조직 구조를 반영해야 생존

## 🚨 흔한 실수

- ❌ 디자인 시스템을 피라미드 꼭대기로 착각 (브랜드가 위)
- ❌ 모든 제품을 하나의 원으로 통합 (도입 상태 차이 무시)
- ❌ 화살표 남발 (수직 배치로 방향성 충분)
- ❌ 다이어그램을 한 번만 그리고 방치 (Living Document 필요)

## 🎯 성공 사례의 공통점

- 중복과 복잡성을 **시각화**하여 경영진 설득
- 시스템 간 관계를 **명확히** 하여 팀 간 협업 개선
- 제품 도입 현황을 **추적**하여 ROI 증명
