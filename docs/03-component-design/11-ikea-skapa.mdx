## 핵심 인사이트

### 1. 80% 커버리지의 법칙
- **목표**: 단일 컴포넌트로 전체 유즈케이스의 80% 커버
- **현실**: IKEA처럼 거대한 조직에서 80%는 매우 야심찬 목표
- **달성 전략**: 최소 필수 요소(title + action)만 강제, 나머지는 옵션으로 설계

### 2. 컴포넌트 ≠ 단일 솔루션
- **초기 가정**: 하나의 카드 컴포넌트로 모든 문제 해결
- **현실**: 3개의 별도 카드 컴포넌트 필요
  - Content Card (풀옵션)
  - Compact Card (경량화)
  - Text Overlay Card (이미지 위 텍스트)
- **교훈**: "유사하지만 목적이 다른" 변형은 별도 컴포넌트로 분리

### 3. 문서화 = 설계 검증 도구
> "문서를 쓰다 보면 설계에서 놓친 부분을 발견한다" - Yunatan (Senior Designer)

- **디자이너 업무 비중**: Figma 20%, 협업/문서화 80%
- **검증 방법**: 다른 사람에게 설명하는 과정에서 결함 발견

## Skapa 시스템 구조

### 규모 지표
| 항목 | 수치 | 의미 |
|------|------|------|
| 팀 규모 | 23명 (Malmo, Amsterdam, Berlin) | 제품-디자인-엔지니어링 통합 팀 |
| 지원 제품팀 | 200+ | 전사 디지털 제품 커버 |
| Figma 에디터 | 400명 | 신규 디자이너는 Skapa 온보딩 필수 |
| Slack 커뮤니티 | 1,700명 | 일일 지원 채널 |
| 연간 처리량 | 50+ 1:1 미팅, 300+ 온보딩 세션, 400~500 요청 | 지원 업무가 일과의 상당 부분 |

### 기술 스택
- **디자인 툴**: Figma
- **프레임워크**: React, Web Components, Vue, Android, iOS (5개 플랫폼 동시 지원)
- **문서화**: Strapi (Headless CMS) + Storybook (플랫폼별)
- **토큰**: 115개 디자인 토큰 (다크모드 포함)
- **아이콘**: 1,000+ 아이콘 (Skapa 팀이 직접 관리)

### 조직 구조의 핵심
- **필수 사용 정책**: Skapa는 "mandatory trade dress" (선택이 아님)
- **도메인 소속**: Experience Platform Capabilities (플랫폼 팀 통합)
- **작은 팀 운영**: 큰 팀 내에서 소규모 풀스택 팀으로 세분화
  - 한 팀이 기능을 처음부터 끝까지 소유
  - 책임감과 신뢰 향상

## 카드 컴포넌트 설계 전략

### 설계 원칙

**최소 필수 요소 (MVP)**:
- **필수**: Title + Action
- **선택**: Badge, Body, Media, Icon
- **목표**: 80% 커버리지 (커스텀 빌드 없이)

**조직 접근 방식**:
1. 전사 카드 변형 조사 (수십 개 발견)
2. Miro 워크샵으로 분류
3. 제품팀 워크샵: "카드란 무엇이고, 무엇이 아닌가?"
4. 외부 벤치마킹
5. Alpha 버전 Figma + 코드 동시 배포

### 세 가지 카드 타입

Skapa는 단일 카드 컴포넌트 대신 **목적별 세 가지 카드**를 제공합니다:

#### 1. Content Card (메인 카드)
- **용도**: 제품 리스팅, 에디토리얼 콘텐츠, 정보 표시
- **구성 요소**:
  - Top: Badge / Icon / Commercial Message (선택)
  - Media Container: 이미지/비디오/Shoppable Markers
  - Content: Title (필수) + Body (선택)
  - Action (필수)
- **특징**: 21개 컬러 테마 지원 (브랜드팀 추가 요구사항)

**구현 예시 (React Compound Component)**:

```tsx
import { Card } from '@ikea/skapa';

function ProductCard() {
  return (
    <Card variant="content" theme="primary" responsive>
      <Card.Media>
        <img src="/product.jpg" alt="Product" />
        <Card.Badge>Sale</Card.Badge>
      </Card.Media>
      <Card.Content>
        <Card.Title>Elegant Living Chair</Card.Title>
        <Card.Body>Modern design with sustainable materials</Card.Body>
        <Card.Action href="/products/chair">
          Shop Now
        </Card.Action>
      </Card.Content>
    </Card>
  );
}
```

**토큰 구조**:

```css
/* Layer 1: Primitive Tokens (색상 팔레트) */
:root {
  --color-theme-primary-base: #0051ba;
  --color-theme-primary-dark: #003d8a;
  --color-theme-primary-light: #cce5ff;
}

/* Layer 2: Semantic Tokens (Content Card 공통) */
:root {
  --card-content-bg: var(--color-white);
  --card-content-border-color: var(--color-gray-200);
  --card-content-title-font-size: 18px;
  --card-content-title-line-height: 1.4;
  --card-content-body-font-size: 14px;
  --card-content-body-color: var(--color-gray-700);
}

/* Layer 3: Component Tokens (Content Card 테마별) */
:root[data-card-theme="primary"] {
  --card-badge-bg: var(--color-theme-primary-base);
  --card-badge-text: var(--color-white);
  --card-action-bg: var(--color-theme-primary-base);
  --card-action-text: var(--color-white);
}

:root[data-card-theme="secondary"] {
  --card-badge-bg: var(--color-gray-200);
  --card-badge-text: var(--color-gray-900);
  --card-action-bg: transparent;
  --card-action-text: var(--color-theme-primary-base);
}
```

#### 2. Compact Card
- **용도**: 밀집 레이아웃, 모바일 그리드, 빠른 네비게이션
- **차별점**: Body 텍스트 없음, 경량화
- **제한**: Shoppable Image 미지원 (작은 사이즈에서 작동 안 함)

**구현 예시**:

```tsx
import { Card } from '@ikea/skapa';

function CompactProductCard() {
  return (
    <Card variant="compact" size="small">
      <Card.Media aspectRatio="1:1">
        <img src="/product-thumb.jpg" alt="Product" />
      </Card.Media>
      <Card.Content>
        <Card.Title>Chair</Card.Title>
        {/* Body 제외 - 자동으로 숨겨짐 */}
        <Card.Action href="/products/chair">
          View
        </Card.Action>
      </Card.Content>
    </Card>
  );
}
```

#### 3. Text Overlay Card
- **용도**: 수직 공간 절약, 모바일 최적화
- **핵심 기능**: 동적 그라디언트 오버레이
  - 텍스트 길이에 따라 그라디언트 자동 확장
  - 항상 충분한 대비 보장 (접근성)

**구현 예시**:

```tsx
import { Card } from '@ikea/skapa';

function ActionCard() {
  return (
    <Card variant="overlay" overlayPosition="bottom">
      <Card.Media aspectRatio="16:9">
        <img src="/campaign.jpg" alt="Campaign" />
        <Card.GradientOverlay
          direction="top"
          intensity="medium"
          minHeight="auto"
        />
      </Card.Media>
      <Card.OverlayContent>
        <Card.Title>Limited Time Offer</Card.Title>
        <Card.Body>Save up to 50% on selected items</Card.Body>
        <Card.Action href="/sale">
          Shop Sale
        </Card.Action>
      </Card.OverlayContent>
    </Card>
  );
}
```

**토큰 구조 (Overlay Card)**:

```css
/* 동적 그라디언트 토큰 */
:root {
  --card-overlay-gradient-start: rgba(0, 0, 0, 0);
  --card-overlay-gradient-end: rgba(0, 0, 0, 0.8);
  --card-overlay-min-height-sm: 40px;
  --card-overlay-min-height-md: 60px;
  --card-overlay-min-height-lg: 80px;
}

/* 텍스트 길이별 자동 확장 */
[data-overlay-text-length="short"] {
  --card-overlay-actual-height: var(--card-overlay-min-height-sm);
}

[data-overlay-text-length="medium"] {
  --card-overlay-actual-height: var(--card-overlay-min-height-md);
}

[data-overlay-text-length="long"] {
  --card-overlay-actual-height: var(--card-overlay-min-height-lg);
}
```

### Media Container 아키텍처

Card.Media는 단순한 이미지 슬롯을 넘어 **동적 인터랙티브 공간**입니다.

**지원되는 콘텐츠 타입**:

| 타입 | Aspect Ratio | 특징 |
|------|-------------|------|
| Image | 1:1, 16:9, 9:16 | IKEA Media Factory 통합 |
| Video | 16:9 | 자체 플레이어 + Transcript 버튼 |
| Commercial Message | 1:1 | 캠페인 배너 |
| Shoppable Markers | 16:9 | 제품 핫스팟 → 직링크 |

**최적화 전략**:
- IKEA Media Factory: 다중 포맷 이미지 자동 제공
- 뷰포트/캐러셀 위치에 따라 최적 aspect ratio 선택
- 응답형 레이아웃에서 자동 전환

**구현 예시**:

```tsx
<Card.Media
  aspectRatio="16:9"
  responsive
  sources={[
    { breakpoint: 'mobile', src: '/img-400w.jpg' },
    { breakpoint: 'tablet', src: '/img-800w.jpg' },
    { breakpoint: 'desktop', src: '/img-1200w.jpg' }
  ]}
>
  <img src="/img-1200w.jpg" alt="Product" />
</Card.Media>
```

## 확장성 아키텍처

### 테마 시스템

Skapa는 **21개 컬러 테마**를 통해 CMS 콘텐츠 크리에이터가 캠페인/이미지에 맞춰 색상을 조정할 수 있습니다.

**구현 패턴**:

```tsx
<Card theme="primary" darkMode>
  {/* 자동으로 다크모드 컬러 적용 */}
</Card>

// 사용 가능한 테마
type CardTheme =
  | 'primary'
  | 'secondary'
  | 'success'
  | 'warning'
  | 'error'
  | 'brand-1' | 'brand-2' // ... up to 21 themes
```

**주의사항**:
- Figma에서 테마 중간 추가 시 변형 수가 기하급수적으로 증가 (1,024개)
- **해결책**: Figma Variables 기능으로 동적 테마 관리 (62개로 축소 예정)

### 반응형 설계

**지원 범위**: 핸드헬드 디바이스 → 75인치 디스플레이

```tsx
<Card
  responsive
  layoutMode="auto"  // 화면에 따라 자동 전환
  // 또는
  layoutMode="locked-vertical"  // 특정 레이아웃 고정
>
  {/* 작은 화면: 세로 스택 */}
  {/* 큰 화면: 이미지-콘텐츠 가로 배치 (Leading/Trailing) */}
</Card>
```

**레이아웃 모드**:
- `auto`: 반응형 자동 전환
- `locked-vertical`: 항상 세로 스택
- `locked-horizontal-leading`: 이미지 좌측 고정
- `locked-horizontal-trailing`: 이미지 우측 고정

### 크로스 플랫폼 배포

**플랫폼 지원**:

| 플랫폼 | 상태 | 월간 다운로드 |
|-------|------|-------------|
| Web | ✅ 80% 채택 (2024-08) | 106,000+ |
| iOS | ✅ 동시 개발 | - |
| Android | ✅ 동시 개발 | - |
| React Native | 계획 중 | - |

**배포 전략**:
- 모든 플랫폼 동시 개발 (선택 아님, 기본 방침)
- 플랫폼별 Storybook 제공
- 크로스 플랫폼 테스트 자동화

## 접근성 아키텍처

카드는 공식 WCAG 패턴이 없으므로 Skapa는 **자체 접근성 전략**을 수립했습니다.

### 키보드 네비게이션

**설계 결정**: 카드 전체를 단일 링크로 처리

```tsx
<Card
  role="link"
  tabIndex={0}
  aria-label="Product card: Elegant Living Chair"
  onKeyDown={handleCardKeydown}
>
  <Card.Media>
    <img src="/product.jpg" alt="" aria-hidden="true" />
  </Card.Media>
  <Card.Content>
    <Card.Title>Elegant Living Chair</Card.Title>
    {/* Body는 스크린 리더에서 건너뜀 */}
    <Card.Body aria-hidden="true">Details...</Card.Body>
    <Card.Action aria-hidden="true">Shop</Card.Action>
  </Card.Content>
</Card>
```

**근거**:
- 10개 카드 캐러셀에서 Title+Body 모두 읽으면 사용자 피로
- 스크린 리더: "Link, [Title]만 발음
- 빠른 탐색 가능

### 테스트 프로세스 (모범 사례)

| 단계 | 담당자 | 시점 |
|------|-------|------|
| 1. 내부 협업 | 시각장애 직원 | Day 1 |
| 2. 전담 검토 | Accessibility Team | 개발 중 |
| 3. 자동 테스트 | 레귤러 CI/CD | 매 커밋 |
| 4. 수동 테스트 | QA | 배포 전 |

**중요**: 접근성은 나중이 아니라 **처음부터** 포함해야 합니다.

## 구현 및 채택 전략

### 얼리 피드백 루프 (필수)

**DO**:
- ✅ Alpha 버전 조기 배포 (Figma + 코드 동시)
- ✅ 주요 사용자(CMS, 제품팀)와 밀접 협업
- ✅ 점진적 채택 수용 (거대 조직에서는 정상)
- ✅ 실제 제품팀 데이터로 검증

**DON'T**:
- ❌ 100% 완벽 후 출시
- ❌ 일방적 배포 (피드백 루프 필수)
- ❌ 마지막에 접근성 추가
- ❌ 문서화 미루기

### Figma 변형 관리

**주의: Compound 속성은 변형 폭발을 초래합니다**

| 시나리오 | 변형 수 | 해결책 |
|---------|-------|--------|
| 색상 5개 × 크기 3개 | 15개 | OK |
| 색상 5개 × 크기 3개 × 레이아웃 3개 | 45개 | OK |
| **색상 21개** × 크기 3개 × 레이아웃 3개 | **189개** | Variables 사용 |
| 테마 추가 시 기하급수 증가 | 1,024+ | 즉시 리팩토링 |

**Content Card 사례**: 개발 중간에 21개 테마 추가 → 1,024개 변형 → 매우 느림

**해결책**: Figma Variables (동적 색상) + 컴포넌트 세트 분리

### 문서화 = 설계 검증

> "문서를 쓰다 보면 설계에서 놓친 부분을 발견한다" - Yunatan (Senior Designer)

**Skapa Hub 지식 허브 구성**:

```
Skapa Hub/
├── 가이드라인/
│   ├── 컴포넌트별 사용 시기
│   └── 금지 패턴
├── 컴포넌트 문서/
│   ├── Card 완전 가이드
│   ├── Props 레퍼런스
│   └── 접근성 체크리스트
├── 파운데이션/
│   ├── 디자인 토큰
│   └── 색상 시스템
├── 디자인 패턴/
├── 플랫폼별 Storybook/
```

**투자 이유**: 문서화 = 사용자 지원 부담 감소 (1,700명 커뮤니티)

### 조직 구조 (성공 요인)

**PedEx 체계** (Product-Engineering-Data-Experience 통합):

```
Design System Team (23명)
├── Product Manager
├── Designers (8명)
│   └── 일일 지원: Slack (1,700명 커뮤니티)
├── Engineers (10명)
│   └── 크로스플랫폼 (Web, iOS, Android, Vue, Web Components)
└── Data & UX Research
```

**성공 요인**:
- 처음부터 모든 직군 참여
- 작은 팀 → 기능 전체 소유 → 책임감 상승
- 풀스택 팀 구성
- 도메인별 책임 명확화

**실패 패턴** (피할 것):
- 디자이너/개발자 분리 작업
- 혼합 프로젝트 배정
- 소유권 부재 (공동 책임 = 책임 없음)

## 체크리스트: 복잡한 컴포넌트 설계

Skapa 카드 설계에서 검증된 10단계 체크리스트입니다.

### Phase 1: 조사 &amp; 정의 (1-2주)

- [ ] **조사**: 전사 변형 수집 (Miro 보드로 분류)
- [ ] **워크샵**: 제품팀과 "X란 무엇이고, 무엇이 아닌가?" 정의
- [ ] **벤치마킹**: 외부 디자인 시스템 3-5개 분석
- [ ] **커버리지**: 80% 목표로 공통 패턴 맵핑

### Phase 2: 설계 (2-4주)

- [ ] **최소화**: 필수 요소만 강제 (Title + Action)
- [ ] **옵션화**: 나머지는 모두 선택 사항
- [ ] **접근성**: Day 1부터 포함 (나중 X)
- [ ] **반응형**: 컴포넌트 내부에 내장 (사용자 몰라도 됨)
- [ ] **테마**: Variables로 동적 관리 (하드코딩 변형 X)

### Phase 3: 개발 &amp; 배포 (4-8주)

- [ ] **병렬화**: 모든 플랫폼 동시 개발 (웹만 먼저 금지)
- [ ] **Alpha**: Figma + 코드 동시 배포 (얼리 피드백)
- [ ] **팀**: 풀스택 구성 (디자이너-개발자 통합)
- [ ] **테스트**: Storybook 라이브 테스트 환경
- [ ] **접근성**: 시각장애 직원 + Accessibility Team 검토

### Phase 4: 문서화 &amp; 출시 (2-4주)

- [ ] **문서**: 작성하며 설계 재검토 (누락 발견)
- [ ] **Props**: 모든 속성 레퍼런스화
- [ ] **반응형**: 레이아웃 동작 명시
- [ ] **A11y**: 접근성 체크리스트 제공
- [ ] **피드백**: 얼리 루프 설정 (주요 사용자)

### Phase 5: 모니터링 &amp; 유지보수

- [ ] **채택률**: 80% 목표로 모니터링
- [ ] **이슈**: 주요 사용자 피드백 수집
- [ ] **점진적**: 대규모 조직 = 느린 채택 정상
- [ ] **정기 테스트**: 장애인 접근성 검증

## 핵심 지표 &amp; 벤치마크

Skapa 프로젝트 규모와 목표:

| 지표 | Skapa | 초기 팀 목표 | 성숙기 목표 |
|------|-------|-----------|-----------|
| **팀 규모** | 23명 (3개 도시) | 5-8명 | 15-20명 |
| **지원 팀** | 200+ 제품팀 | 10-20팀 | 50-100팀 |
| **컴포넌트** | 60+ | 10-15개 | 30-50개 |
| **플랫폼** | 5개 (Web, iOS, Android, Vue, Web Components) | 1개 (웹) | 2-3개 |
| **채택률** | 80% (2024-08) | 50% (Month 6) | 80-90% |
| **문서화 시간** | 디자이너 80% | 50-60% | 70-80% |
| **Slack 커뮤니티** | 1,700명 | 100-200명 | 500-1,000명 |

## 성공 사례 요약

**Skapa Card 설계의 핵심 성공 요인**:

1. **MVP 마인드셋**: 필수 요소만 강제 (Title + Action) → 80% 커버리지 달성
2. **목적별 분리**: 단일 "Card" 대신 3가지 타입 (Content, Compact, Overlay)
3. **조직 통합**: PedEx 체계로 모든 직군 Day 1부터 참여
4. **얼리 피드백**: Alpha 배포 (Figma + 코드) → 빠른 피드백 루프
5. **접근성 우선**: 나중이 아닌 처음부터 시각장애인 협업
6. **문서화 투자**: 80% 문서화 = 1,700명 커뮤니티 자립

> **실제 사례**: IKEA Skåpa는 40개 국가, 200개 디지털 터치포인트를 통합하는 엔터프라이즈급 디자인 시스템입니다. 스웨덴 본사 15명 팀이 관리하며, 토큰 기반 아키텍처로 국가별 브랜드 조정(색상, 타이포그래피)을 자동화하여, 글로벌 일관성과 로컬 유연성을 동시에 달성합니다.

> **심화 이론**: 엔터프라이즈 디자인 시스템은 **Conway's Law**를 역으로 활용한 사례입니다. "조직 구조가 시스템 구조를 결정한다"는 법칙을 이해하고, 중앙 팀(Core)과 도메인 팀(Contributors)의 책임을 명확히 분리하여, 시스템 아키텍처가 조직 커뮤니케이션을 강제하도록 설계합니다. 이는 Team Topologies의 "Platform Team" 개념을 디자인에 적용한 것입니다.
