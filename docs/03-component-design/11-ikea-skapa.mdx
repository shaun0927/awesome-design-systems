## 핵심 인사이트

### 1. 80% 커버리지의 법칙
- **목표**: 단일 컴포넌트로 전체 유즈케이스의 80% 커버
- **현실**: IKEA처럼 거대한 조직에서 80%는 매우 야심찬 목표
- **달성 전략**: 최소 필수 요소(title + action)만 강제, 나머지는 옵션으로 설계

### 2. 컴포넌트 ≠ 단일 솔루션
- **초기 가정**: 하나의 카드 컴포넌트로 모든 문제 해결
- **현실**: 3개의 별도 카드 컴포넌트 필요
  - Content Card (풀옵션)
  - Compact Card (경량화)
  - Text Overlay Card (이미지 위 텍스트)
- **교훈**: "유사하지만 목적이 다른" 변형은 별도 컴포넌트로 분리

### 3. 문서화 = 설계 검증 도구
> "문서를 쓰다 보면 설계에서 놓친 부분을 발견한다" - Yunatan (Senior Designer)

- **디자이너 업무 비중**: Figma 20%, 협업/문서화 80%
- **검증 방법**: 다른 사람에게 설명하는 과정에서 결함 발견

## Skapa 시스템 구조

### 규모 지표
| 항목 | 수치 | 의미 |
|------|------|------|
| 팀 규모 | 23명 (Malmo, Amsterdam, Berlin) | 제품-디자인-엔지니어링 통합 팀 |
| 지원 제품팀 | 200+ | 전사 디지털 제품 커버 |
| Figma 에디터 | 400명 | 신규 디자이너는 Skapa 온보딩 필수 |
| Slack 커뮤니티 | 1,700명 | 일일 지원 채널 |
| 연간 처리량 | 50+ 1:1 미팅, 300+ 온보딩 세션, 400~500 요청 | 지원 업무가 일과의 상당 부분 |

### 기술 스택
- **디자인 툴**: Figma
- **프레임워크**: React, Web Components, Vue, Android, iOS (5개 플랫폼 동시 지원)
- **문서화**: Strapi (Headless CMS) + Storybook (플랫폼별)
- **토큰**: 115개 디자인 토큰 (다크모드 포함)
- **아이콘**: 1,000+ 아이콘 (Skapa 팀이 직접 관리)

### 조직 구조의 핵심
- **필수 사용 정책**: Skapa는 "mandatory trade dress" (선택이 아님)
- **도메인 소속**: Experience Platform Capabilities (플랫폼 팀 통합)
- **작은 팀 운영**: 큰 팀 내에서 소규모 풀스택 팀으로 세분화
  - 한 팀이 기능을 처음부터 끝까지 소유
  - 책임감과 신뢰 향상

## 카드 컴포넌트 설계 프로세스

### Phase 1: 현실 파악 (Reality Check)
```
문제 상황:
- 카드가 IKEA 전 제품에 산재
- 각 팀마다 "카드"의 정의가 다름
- 접근성 미준수 다수 발견
- 기존 컴포넌트는 유연성 부족
```

**액션**:
1. 전사 카드 변형 조사 (수십 개 발견)
2. Miro 워크샵으로 분류 작업
3. 제품팀과 정의 워크샵: "카드란 무엇이고, 무엇이 아닌가?"
4. 실제 사례 외부 벤치마킹

### Phase 2: 요구사항 정의
**핵심 결정**:
- **최소 필수 요소**: Title + Action만 강제
- **목표 커버리지**: 80% (커스텀 빌드 없이)
- **얼리 피드백 루프**: Alpha 버전을 Figma + 코드로 동시 배포

**도출된 3가지 카드**:

#### 1. Content Card (메인 카드)
- **용도**: 제품 리스팅, 에디토리얼 콘텐츠, 정보 표시
- **구성 요소**:
  - Top: Badge / Icon / Commercial Message (선택)
  - Media Container: 이미지/비디오/Shoppable Markers
  - Content: Title (필수) + Body (선택)
  - Action (필수)
- **특징**: 21개 컬러 테마 지원 (브랜드팀 추가 요구사항)

#### 2. Compact Card
- **용도**: 밀집 레이아웃, 모바일 그리드, 빠른 네비게이션
- **차별점**: Body 텍스트 없음, 경량화
- **제한**: Shoppable Image 미지원 (작은 사이즈에서 작동 안 함)

#### 3. Text Overlay Card
- **용도**: 수직 공간 절약, 모바일 최적화
- **핵심 기능**: 동적 그라디언트 오버레이
  - 텍스트 길이에 따라 그라디언트 자동 확장
  - 항상 충분한 대비 보장 (접근성)

### Phase 3: Media Container 혁신
**기존**: 정적 이미지 슬롯
**개선**: 동적 인터랙티브 공간

지원 콘텐츠:
- 이미지 (다양한 aspect ratio: 1:1, 16:9, 9:16)
- 비디오 (자체 플레이어 + 최근 transcript 버튼 추가)
- Commercial Messages
- Shoppable Markers (제품 페이지 직링크)

**연동 시스템**:
- IKEA Media Factory에서 다중 포맷 이미지 자동 제공
- 뷰포트/캐러셀 위치에 따라 최적 aspect ratio 선택

## 확장성 전략

### 1. 테마 시스템
- **21개 컬러 테마**: 브랜드팀 요구사항 (개발 중간에 추가!)
- **다크/라이트 모드**: 각 카드가 테마 파일 포함
- **자동 텍스트 색상 조정**: 테마 전환 시 가독성 보장
- **용도**: CMS 콘텐츠 크리에이터가 캠페인/이미지에 맞춰 색상 조정

### 2. 반응형 설계
- **지원 범위**: 소형 핸드헬드 디바이스 → 75인치 디스플레이
- **자동 레이아웃 전환**:
  - 작은 화면: 세로 스택
  - 큰 화면: 이미지-콘텐츠 가로 배치 (Leading/Trailing 선택 가능)
- **잠금 옵션**: 특정 레이아웃 고정 가능

### 3. 크로스 플랫폼
**기존 카드**: 웹만 지원, 106,000 월간 다운로드
**신규 카드**: iOS + Android 추가 → 진정한 크로스 플랫폼

**배포 전략**:
- 모든 플랫폼 동시 개발 (Skapa의 기본 방침)
- 플랫폼별 Storybook 제공
- 2024년 8월: 웹 80% 채택률 달성

## 접근성 구현 (No WCAG Pattern)

### 문제: 카드는 공식 WCAG 패턴이 없음
**해결**: 자체 접근성 전략 수립

### 키보드 네비게이션 설계
1. **카드 전체를 단일 링크로 처리**
   - 스크린 리더: "Link, [Title]" 발음
   - Body 텍스트는 읽지 않음
2. **이유**: 10개 카드 캐러셀에서 Title+Body 모두 읽으면 사용자 피로
3. **결과**: 빠른 탐색 가능

### 테스트 프로세스
- **내부 시각장애 직원과 협업**
- **전담 Accessibility Team 리뷰**
- **Day 1부터 포함** (나중이 아님)

## 실전 팁 & 교훈

### 채택 관련
```
DO:
- Alpha 버전 조기 배포 (Figma + 코드 동시)
- 주요 사용자(CMS, 제품팀)와 밀접 협업
- 점진적 채택 수용 (거대 조직에서는 정상)

DON'T:
- 100% 완벽 후 출시
- 일방적 배포 (피드백 루프 필수)
```

### 변형 관리
**Figma의 함정**:
- Content Card 현재 상태: **1,024개 변형**
- 원인: 21개 컬러 테마 중간 추가
- 결과: 느리고 유지보수 지옥
- **해결 예정**: 새 Variables 기능으로 62개로 축소

**교훈**: 테마 시스템은 Variables 같은 동적 방식으로 설계해야 함

### 문서화 우선순위
> "디자이너로서 내 시간의 80%는 협업과 문서화다" - Yunatan

**Skapa Hub (지식 허브) 구성**:
- 가이드라인
- 컴포넌트 문서
- 파운데이션
- 디자인 패턴
- 플랫폼별 Storybook

**투자 이유**: 문서화 = 사용자 지원 부담 감소

### 조직 구조
```
성공 요인:
- PedEx 체계 (Product-Engineering-Data-Experience 통합)
- 처음부터 모든 직군 참여
- 작은 팀 → 기능 전체 소유 → 책임감 상승
- 풀스택 팀 구성

실패 요인 (과거):
- 디자이너/개발자 분리 작업
- 혼합 프로젝트 배정
- 소유권 부재
```

### AI 챗봇 지원
**현재 상태**: 없음 (Spotify에 조언 요청 중)
**필요성**: 1,700명 커뮤니티 지원 부담 완화

## 체크리스트 - 복잡한 컴포넌트 설계 시

### 사전 조사 단계
- [ ] 전사 변형 조사 (최대한 많이 수집)
- [ ] 제품팀과 정의 워크샵 ("X란 무엇이고, 무엇이 아닌가?")
- [ ] 외부 벤치마킹 (다른 디자인 시스템 참고)
- [ ] 공통 패턴 맵핑 (80% 커버리지 목표)

### 설계 단계
- [ ] 최소 필수 요소만 강제 (Title + Action)
- [ ] 나머지는 옵션으로 설계
- [ ] 접근성을 처음부터 고려 (Day 1)
- [ ] 반응형을 컴포넌트 내부에 내장 (사용자가 신경 안 쓰게)

### 개발 단계
- [ ] 모든 플랫폼 동시 개발 (웹만 먼저 X)
- [ ] Alpha 버전 조기 배포 (Figma + 코드)
- [ ] Storybook으로 라이브 테스트 환경 제공
- [ ] 풀스택 팀 구성 (디자이너-개발자 통합)

### 문서화 단계
- [ ] 문서 작성하며 설계 재검토
- [ ] 속성(Properties) 전부 문서화
- [ ] 반응형 동작 명시
- [ ] 접근성 가이드 포함

### 배포 단계
- [ ] 얼리 피드백 루프 설정
- [ ] 주요 사용자 그룹과 협업 (CMS, 주요 제품팀)
- [ ] 점진적 채택 계획 수립
- [ ] 채택률 모니터링 (목표: 80%)

### 유지보수 단계
- [ ] Variables로 테마 관리 (하드코딩 변형 X)
- [ ] 정기 내부 장애인 테스트
- [ ] 플랫폼 간 동기화 유지
