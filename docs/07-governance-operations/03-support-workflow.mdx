## 디자인 시스템 지원 워크플로우

> Original issue: shaun0927/stocktitan-crawler#567

## 📌 핵심 개념

### 1. Support의 정의
> 디자인 시스템이 고객의 요청과 문제를 건별로 접수(intake), 분류(triage), 라우팅(route), 할당(assign), 해결(resolve)하는 방식

**핵심 특징**:
- **인바운드 방향**: 사용자가 먼저 요청을 시작
- **개별적 관계**: 1대1 문제 해결
- **즉각적 응답 기대**: 도움이 필요한 순간 차단될 수 있음

### 2. Support가 중요한 이유
**Morningstar 사례 (2017년 6월)**:
- 출시 직후 8주간 **100개 이상의 케이스** 발생
- Slack, 이메일, 티켓 시스템 등 여러 채널에서 동시다발적 요청
- 프로세스를 구축하며 신뢰 확보

**핵심 교훈**:
```
디자인 시스템은 긴급하고 지속적으로 발생하는 디자이너와 개발자의
요구사항을 해결할 준비가 되어 있어야 한다.
이는 사전 투자와 지속적인 운영 commitment가 필요하며,
결코 가볍게 넘길 수 없는 영역이다.
```

## 🎯 실무 노하우

### 1. 케이스 유형별 대응 전략

#### A. 즉시 대응 필요 (Immediate Response)
| 케이스 유형 | 설명 | 대응 시간 | 이유 |
|------------|------|----------|------|
| **Get Help** | 시스템 기능 사용법 질문 | 즉시 | 작업 차단 가능 |
| **Defect** | 버그, 예상치 못한 동작 | 즉시 | 프로덕션 영향 |
| **Review Request** | 규정 준수 검토 요청 | 즉시 | 릴리스 차단 가능 |

#### B. 숙고 필요 (Requires Deliberation)
| 케이스 유형 | 설명 | 질문 포인트 |
|------------|------|------------|
| **Feature Request** | 시스템에 없는 기능 요청 | "언제까지 필요한가?" |
| **Contribution** | 외부 기여 제안 | "다른 팀도 필요한가?" |
| **Training** | 교육/온보딩 요청 | "개인? 팀? 대규모?" |

### 2. 채널 전략

#### 공식 채널
| 채널 | 장점 | 단점 | 최적 용도 |
|------|------|------|----------|
| **Chat (Slack, Teams)** | 가장 빠른 응답, 선호도 높음 | 추적 어려움 | Help, 간단한 질문 |
| **Task Management (Jira)** | 추적 용이, 워크플로우 통합 | 진입장벽 높음 | Defect, Feature |
| **Email (시스템 전용)** | 공식적, 기록 남음 | 느림 | 공식 요청 |

#### 비공식 채널 주의
- Personal chat/email, Design critique, Code review 중 발생
- 적절한 공식 채널로 안내, 최소한의 정보만 수집

### 3. 데이터 수집 전략

#### 필수 데이터 (All Cases)
```markdown
✓ Requester 정보 (이름, 이메일, 팀/제품/조직)
✓ Type (Defect, Help, Review Request, Feature, Contribution)
✓ Description (자유 텍스트)
✓ Attachments (스크린샷, 제안 디자인)
✓ Channel (Slack, Jira, Email 등)
```

#### Defect 전용 추가 데이터
```markdown
✓ What's experienced (스크린샷, 에러 메시지)
✓ Steps to Reproduce (재현 단계)
✓ Versions (브라우저, 디자인 시스템 버전)
```

#### Feature Request/Contribution 전용
**Keep it Light!**
```markdown
✓ Type (새로운 것? 기존 것의 개선?)
✓ Shared Need (다른 팀도 필요한가?)
✓ Urgency (이번 주? 이번 달? 이번 분기?)
```

### 4. 기대치 관리
**잘못된 기대**: "다음 주까지 프로덕션에 배포"
**올바른 응답**: "긴급성을 고려하면, 제품 내에서 로컬로 만드는 게 최선입니다."

**핵심 원칙**:
```
디자인 시스템은 장기적 관점에서 운영된다.
빠른 응답 워크플로우 ≠ 복잡한 기능을 하루 만에 출시
```

### 5. Triage & Routing

#### Triage 장소와 시간
- Daily scrum/standup
- Sprint backlog
- Weekly planning queue
- **"To Triage" / "Not Yet Prioritized" Kanban 컬럼** (저자 선호)

#### 즉시 해결 가능한 케이스
- 회의 중 빠른 답변, Slack 채팅 몇 마디, 간단한 이메일 회신
- 공식 추적 불필요
- **하지만**: 디자이너/개발자의 일상 업무 중 20~50%를 차지할 수 있음

## 📊 지원 워크플로우 상세

### 1. SLA (Service Level Agreement) 설정
**응답 시간 vs 해결 시간**:
- **First Response**: "들었습니다, 처리 중입니다" → 신뢰 구축
- **Resolution**: "케이스가 해결되었습니다" → 실제 문제 완전 해결

**설문 결과**: 대부분의 팀은 **1일 이내 응답 약속**

### 2. 역할 분담 (Roles)

**옵션 1: 한 사람** (효율적, 단일 장애점)
**옵션 2: 정의된 팀 서브셋** (백업 가능, 조율 필요)
**옵션 3: 팀 전원** (가장 빠른 응답, 혼돈 위험)

**저자 추천**: 소수를 워크플로우 운영에 배정, 백채널 조율, 순환(tour-of-duty) 고려

#### 지원이 모두에게 맞는 것은 아니다
- 지원을 좋아하는 사람: 문제 해결의 왕복을 즐김, 연결성에서 긍정적 느낌
- 지원을 싫어하는 사람: 인내심, 호기심 부족, Context switching이 방해

#### "The Swarm" 조심
- 2명 이상이 동시에 응답하면 Crosstalk, 상충되는 조언 발생
- 해결책: 비공개 백채널에서 팀원들이 조율 후 응답

### 3. 도구와 가시성
**원칙**: 도구는 요청자, 응답자, 프로세스 관리자 간 가시성과 협업을 허용해야 함

**도구 성숙도 단계**:
```
1단계: 도구 없음 (팀 규모가 작을 때)
2단계: 스프레드시트 (Excel, Google Sheets)
3단계: Task management (Jira, Asana, Github Issues) ← 대부분 여기
4단계: Ticketing software (Jira Service Desk) ← 대량 케이스 처리
```

## ✅ 실무 체크리스트

### 지원 시작 전 준비사항
```markdown
□ 지원할 케이스 유형 정의
□ 공식 채널 선택 (너무 많지 않게)
□ 비공식 채널 처리 방법 정의
□ 수집할 데이터 정의
□ SLA 결정 및 공개
□ Triage 시간/장소 결정
□ 역할 분담
□ 도구 선택
□ 팀원 개별 면담
```

### 지원 운영 중 점검사항
```markdown
□ 응답 시간 SLA 준수 확인
□ 채널별 모니터링
□ Triage 프로세스 실행
□ 응답 품질 관리 ("The Swarm" 방지)
□ 즉시 해결 케이스 인정
□ 데이터 추적 및 분석
□ 팀원 번아웃 방지
```

## 💡 핵심 Takeaways

1. **지원은 선택이 아닌 필수** - 신뢰 = 빠르고 품질 높은 지원
2. **사전 투자와 지속적 commitment 필요**
3. **채널은 신중하게 선택** - 너무 많으면 관리 불가
4. **기대치 관리** - 모든 케이스가 긴급한 것은 아님
5. **케이스 유형별 차별화** - Help/Defect → 즉시, Feature → 숙고
6. **SLA 공개** - 투명성이 신뢰 구축
7. **즉시 해결 케이스 인정** - 시간 소요 인정하고 계획
8. **역할 분담의 균형** - 한 사람 또는 정의된 서브셋 추천
9. **"The Swarm" 방지** - 백채널 조율 후 응답
10. **도구는 가시성 중심** - 대부분 Task Management로 충분
