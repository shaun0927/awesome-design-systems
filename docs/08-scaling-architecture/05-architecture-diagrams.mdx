## 시각적 심볼 체계

| 심볼 | 의미 | 설명 |
|------|------|------|
| ◆ (다이아몬드) | 디자인 시스템 | 비주얼 스타일, UI 컴포넌트, 코드 자산 |
| ● (원형) | 제품/경험 | 디지털 경험 또는 그 일부 |
| ■ (정사각형) | 브랜드 | 색상, 타이포그래피, 일러스트레이션 |
| D (빨강) | 디자인 에셋 | Figma 라이브러리 |
| C (파랑) | 코드 라이브러리 | React, HTML 패키지 |

## 핵심 개념

### 문제 정의
대부분의 기업은 "하나의 디자인 시스템이 모든 제품을 지원"하는 단순한 구조를 넘어선 **복잡한 생태계**를 가지고 있다. 이러한 복잡한 시스템의 폭과 다양한 고객에게 제공하는 기능을 빠르게 보여주고 가르칠 수 있는 **시각적 어휘가 부족**하다.

### 솔루션: 시각적 심볼 체계
**브랜드(Brands), 시스템(Systems), 제품(Products)을 연결하는 시각적 어휘**

## 아키텍처 다이어그램 유형

### 1. 단순형 (Simple) - 스타트업, 단일 제품

```mermaid
graph TD
    B["Brand\nCompany Identity"]
    S["Design System\n(Figma + React)"]
    P1["Product A"]
    P2["Product B"]
    P3["Product C"]

    B --> S
    S --> P1
    S --> P2
    S --> P3

    style B fill:#ffd43b,color:#000
    style S fill:#339af0,color:#fff
    style P1 fill:#51cf66,color:#fff
    style P2 fill:#51cf66,color:#fff
    style P3 fill:#51cf66,color:#fff
```

**npm 의존성 트리:**

```
@company/product-a
  └── @company/design-system@3.2.1

@company/product-b
  └── @company/design-system@3.2.1

@company/product-c
  └── @company/design-system@3.1.0  ← 아직 이전 버전
```

### 2. 중개형 (Intermediary) - 프레임워크 번역 레이어

```mermaid
graph TD
    S["Core System\n(HTML/CSS)"]
    R["React\nTranslation"]
    V["Vue\nTranslation"]
    D["Direct\nAdoption"]

    P1["React App"]
    P2["Vue App"]
    P3["Static Site"]

    S --> R
    S --> V
    S --> D
    R --> P1
    V --> P2
    D --> P3

    style S fill:#339af0,color:#fff
    style R fill:#cc5de8,color:#fff
    style V fill:#51cf66,color:#fff
```

**npm 의존성 트리:**

```
@company/react-app
  └── @company/react-ui@2.0.0
       └── @company/core-css@3.2.1

@company/vue-app
  └── @company/vue-ui@1.5.0
       └── @company/core-css@3.2.1
```

## 다이어그램 작성 가이드

### 작성 전 질문

- [ ] 우리 조직에는 몇 개의 디자인 시스템이 있는가?
- [ ] 각 시스템은 어떤 제품을 지원하는가?
- [ ] 브랜드 아이덴티티는 어디서 관리되는가?
- [ ] 디자인 자산(D)과 코드(C) 중 무엇을 제공하는가?
- [ ] 제품별 시스템 도입 상태는?
- [ ] 조직 경계는 어떻게 나뉘는가?

### 유지보수 주기

| 주기 | 업데이트 내용 |
|------|------------|
| **분기별** | 제품 도입 상태 업데이트 |
| **반기별** | 시스템 기능 검토 |
| **연간** | 조직 구조 변경 반영 |

### 흔한 실수

- 디자인 시스템을 피라미드 꼭대기로 착각 (브랜드가 위)
- 모든 제품을 하나의 원으로 통합 (도입 상태 차이 무시)
- 화살표 남발 (수직 배치로 방향성 충분)
- 다이어그램을 한 번만 그리고 방치 (Living Document 필요)

## 실무 노하우

### 다이어그램 활용
- **경영진 보고**: 시스템 투자 정당화
- **팀 온보딩**: 새 멤버에게 생태계 설명
- **로드맵 계획**: 확장 우선순위 결정
- **중복 식별**: 불필요한 시스템 발견
- **인수합병**: 통합 시나리오 시뮬레이션

### 3가지 황금률

1. **단순성 추구**: 복잡한 생태계를 한눈에
2. **상태 명확화**: 제품 도입 상태를 색상으로 즉시 파악
3. **조직 반영**: 실제 조직 구조를 반영해야 생존
