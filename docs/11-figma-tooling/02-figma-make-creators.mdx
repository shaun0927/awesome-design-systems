## Figma Make 크리에이터 제작 프로세스 정리 (일본어 원본)

> **출처**: Made with Figma Make - 크리에이터가 말하는 제작 프로세스 세션 (일본어 자동 생성 자막)  
> **발표자**: 구와하라 (ワンパク), 미야모토 (IE3/DNA)  
> **이벤트**: Figma Make 캠페인 (2024년 11월, 시부야 스크램블 교차점 광고 + 하라주쿠 팝업)

---

## 📌 핵심 인사이트

### 1. **Figma Make = 아이디어를 즉시 형태로 만드는 도구**
- **기존 프로토타이핑의 한계**: 아이디어 → 프로토타입 제작에 2주~1개월 소요, 검증 가능한 아이디어 수 제한
- **Figma Make의 혁신**: 텍스트 프롬프트만으로 인터랙티브 앱 생성, 디자인 데이터를 함께 전달하면 재현성 높은 UI 구현
- **핵심 메시지**: "아이디어의 씨앗 → 제품"까지의 거리를 극적으로 단축

### 2. **AI와의 협업 = 상호 존중의 대화**
- 구와하라: "후반에는 Figma Make에 **경어**로 말하게 됨. '감사합니다, 다음은 이걸 부탁합니다' 식으로"
- 미야모토: "잘 나온 걸 **칭찬**하며 진행. '좋네요, 0.5초 후에 시작해봐요' 같은 피드백"
- → **AI를 도구가 아닌 '동료(相棒)'로 인식**하는 순간 생산성이 급상승

### 3. **제약(Constraint)이 창의성을 높인다**
- 미야모토: "**자연어로만 제작**하는 제약을 스스로 설정 → AI가 만든 '멋있음'을 존중하며 협업"
- 구와하라: "카메라 각도를 일본어로 설명하다 한계 → **디버거 UI를 Figma Make에 만들게 해서** 수치로 소통"
- → 제약 속에서 새로운 작업 방식(Tools for Making) 발견

## 🎨 크리에이터별 제작 프로세스

### 작품 1: **Autumn Walk (오텀 워크)** - 구와하라 (ワンパク)

#### 컨셉
- **모티프**: 낙엽을 발로 밟으며 걷는 감각 → 3D 공간에서 재현
- **타겟**: 시부야 교차점 (해외 관광객 고려 → 일본의 단풍 문화 표현)
- **인터랙션**: 십자키로 3D 공간 이동 + 발걸음에 따라 낙엽이 물리 엔진으로 날아오름

#### 제작 과정 (226개 버전 / 1~2일 집중 작업)

**Phase 1: Dirty Start (더럽게 시작하기)**
- **첫 프롬프트**: "3D 공간에서 알록달록한 낙엽을 발로 밟으며 물리 엔진으로 날아오르게, 카메라는 로우앵글로 발밑 촬영, 시네마틱한 깊이감"
- → ChatGPT로 프롬프트 정제 후 한 번에 투입 → **1회 시도로 기본 개념 구현 성공**
- 문제점: 움직임이 어색하고 무거움 (2D 캔버스 사용)

**Phase 2: Step by Step (단계별 개선)**
1. **스켈레톤 구축**: "십자키로 보행 가능한 메타버스 만들기 + Three.js 사용" → 3D 환경 기반 완성
2. **신발 모델링 추가**: GLB 파일 임포트 → **별도 Model Viewer 제작**하여 애니메이션 단독 검증
3. **카메라 앵글 조정**: 
   - 초기: "조금 더 발에 가깝게" 같은 일본어 지시 → 정확도 낮음
   - 해결: **카메라 파라미터 디버거 UI 생성** → 수치로 소통 (거리, 각도, 줌 비율 표시)
4. **낙엽 텍스처**: Figma Design에서 AI로 낙엽 1장 생성 → 6가지 색상/형태 변형 → 3D 공간에 파티클로 배치
5. **라이팅 게이지**: 프로덕션 팀의 색감 조정용 인터페이스 추가

**Phase 3: Tools for Making (제작 도구 제작)**
- GLB 체커, 카메라 디버거, 조명 조정 패널 등 → **모두 Figma Make가 생성한 보조 기능**
- → "더 나은 제작을 위한 도구를 AI에게 만들게 하는" 메타적 접근

#### 3가지 학습

| 원칙 | 의미 | 적용 예시 |
|------|------|-----------|
| **Start Dirty** | 깔끔하게 시작하지 말고 빠르게 완성형 이미지 확인 | 첫 프롬프트로 거친 프로토타입 → 작업 의욕 상승 |
| **Step by Step** | 한 번에 모든 걸 전달하지 말고 점진적 축적 | 메타버스 기반 → 신발 → 카메라 → 낙엽 순차 구축 |
| **Tools for Making** | 더 나은 제작을 위한 도구를 AI에게 요청 | 디버거, 뷰어 같은 메타 도구로 정밀 제어 |

### 작품 2: **Party Type (파티 타입)** - 미야모토 (IE3/DNA)

#### 컨셉
- **모티프**: 시부야 교차점 = 다양한 국적이 교차하는 장소 → "인사" 메시지를 파티클로 표현
- **기술**: WebGL 파티클 표현 + 실시간 텍스트 입력 + 배경 스트라이프 조정
- **특징**: 폰트 변경, 컬러 파라미터 조정 등 인터랙티브 요소

#### 제작 과정 (약 100회 프롬프트 왕복 + 팀원 협업)

**Phase 1: Scrap and Build (고속 스케치)**
- **실험 1**: 과거 프로토타입 디자인 → Figma Make에 전달 → "비슷하지만 완전 재현은 안 됨" (배치는 되지만 디테일 손실)
- **실험 2**: 모션 그래픽 스튜디오 툴 → "UI는 생성되지만 버튼이 작동 안 함" (외형만 구현)
- **실험 3**: 키보드 타이핑 → 비주얼라이즈 → "인터랙션성 부족, 자연어로 비주얼 설명 어려움"
- **실험 4**: SVG 제너레이티브 그래픽 → "나선형, 슈퍼 타원 생성기 완성" (디자이너에게 유용하지만 너무 지미함)
- **실험 5**: 파티클 비주얼라이저 → **"이 방향성으로 가능할 것 같다!"** → 텍스트 입력 + 파티클 표현으로 확정

**Phase 2: 자연어 제약 + 칭찬 피드백**
- **제약 설정**: "코드를 건드리지 않고 자연어로만 제작" → AI의 '멋있음' 감각 존중
- **프롬프트 예시**: 
  - "사용자가 문자를 입력할 수 있게 해줘"
  - "슬라이더를 **멋있게** 만들어줘" (추상적 지시 → AI의 해석 존중)
  - "좋네요! 0.5초 후에 시작해봐요" (칭찬 + 다음 단계 유도)
- → **1개 기능씩 추가/개선** (대충 지시하면 기존 기능 파괴됨)

**Phase 3: 팀원 협업 (Pair Coding)**
- **1차 드래프트 (100회 왕복)**: 기본 기능 완성 → 하지만 파라미터 미세 조정, 마우스 인터랙션 한계
- **문제**: "자연어로는 파티클 분위기, 깊이감 조정이 의사소통 어려움"
- **해결**: 팀원(타나카)에게 React 코드 작성 의뢰 → 파라미터 조정 + 마우스 인터랙션 추가
- **2차 문제**: "배경이 검은색이면 교차점에서 눈에 안 띔" → 그라데이션 요청했으나 Figma Make 실패
- **2차 해결**: 타나카가 배경 셰이더 + 이펙트 추가 → 최종 완성

#### 제작 철학
- **AI 역할**: "50점짜리 베이스를 1분 만에 생성" → 인간이 개성/공격(攻撃) 추가
- **Scrap and Build 가속화**: 기존 1일 단위 → **1프롬프트(1분) 단위**로 "만들고 버리고" 반복 가능
- → "AI가 뭘 할 수 있는지 바이어스 없이 시도" → 가능성 발견 후 브러시업

## 🛠️ Figma Make 실전 워크플로우

### 1. **프롬프트 전략**

#### 초기 단계
```
❌ 나쁜 예: "3D 낙엽 앱 만들어줘"
✅ 좋은 예: "3D 공간에서 알록달록한 낙엽이 물리 엔진으로 날아오르는 인터랙션, 
            카메라는 로우앵글(발밑 시점), Three.js 사용, 십자키로 이동 가능"
```

#### 개선 단계
```
✅ "이것은 구현 지시가 아닙니다. 이걸 하고 싶은데 어떻게 전달하면 좋을까요?"
→ Figma Make가 일본어로 상담 모드로 전환 (실제 코드 변경 없음)

✅ "이 부분은 변경하지 말고, 여기만 수정해주세요"
→ 예상치 못한 변경 방지
```

#### 협업 모드
```
✅ "감사합니다. 잘 되었네요. 다음은 이걸 부탁합니다."
✅ "좋네요! 0.5초 후에 시작해봐요."
→ 경어 + 칭찬 = 작업자의 심리적 만족도 향상 (AI와의 감정적 연결)
```

### 2. **외부 파일 가져오기 (GLB 예시)**

| 방법 | 장단점 |
|------|--------|
| **Figma Make 내 Importer 생성** | ✅ UI 내에서 드래그 드롭 가능<br/>❌ 엔드유저용 업로드로 변질될 수 있음 |
| **GitHub Pages 호스팅** | ✅ 고정 파일을 안정적으로 참조<br/>✅ 풀 패스(full path)로 접근 가능 |
| Google Drive | ❌ CORS 에러, 도메인 접근 제한 실패 |

**구와하라의 최종 솔루션**: 
```
1. GitHub 리포지토리에 GLB 전용 폴더 생성
2. GitHub Pages로 호스팅
3. Figma Make에 "이 폴더의 shoe.glb 파일 사용"이라고 풀 패스 전달
```

### 3. **컨트롤 패널 생성**

#### 구와하라 방식 (디버거 우선)
```
프롬프트: "카메라 각도, 거리, 줌 비율을 실시간으로 보여주는 디버거 만들어줘"
→ 검은 박스에 수치 표시 → 마우스로 조정하며 적정값 찾기 → "기본값을 이 수치로 설정"
```

#### 미야모토 방식 (UI 구조 우선)
```
프롬프트: "텍스트 입력 화면과 파라미터 설정 화면을 분리해줘"
→ 출력 확인 후: "폰트 선택 추가", "컬러 파라미터 추가" 같이 점진적 요청
```

### 4. **환경 구축 (3D, WebGL 등)**

```
✅ 간단한 방법: "3D 공간 만들어줘" → Figma Make가 알아서 Three.js 등 선택
✅ 명시적 방법: "Three.js로 3D 메타버스 만들어줘"
✅ 대안 제시: "Unreal Engine 쓸 수 있어?" 같은 질문도 가능

→ 환경 구축은 100% Figma Make가 자동 처리 (사전 설정 불필요)
```

## ✅ 실전 체크리스트

### 프로젝트 시작 전
- [ ] **Dirty Prototype 먼저**: 완벽한 첫 프롬프트보다 빠른 결과물로 동기 부여
- [ ] **AI 가능성 탐색**: 바이어스 없이 "이거 되나?" 시도 (Scrap and Build 고속화)
- [ ] **제약 설정**: "자연어로만", "이 부분은 변경 금지" 같은 룰 명시

### 제작 중
- [ ] **1기능 1프롬프트**: 한 번에 여러 요청 → 기존 기능 파괴 위험
- [ ] **상담 모드 활용**: "이건 구현이 아니에요. 어떻게 하면 좋을까요?"
- [ ] **Tools for Making**: 디버거, 뷰어 같은 메타 도구 요청
- [ ] **칭찬과 감사**: "좋네요", "감사합니다" → 심리적 협업 품질 향상

### 한계 지점
- [ ] **자연어 한계 인지**: 수치, 파라미터 미세 조정은 전문가 협업 고려
- [ ] **Figma Design 활용**: 구체적인 비주얼(로고, 레이아웃)은 디자인 파일로 전달
- [ ] **코드 개입 시점**: React 컴포넌트 작성 → Figma Make로 재반입 가능

## 🎯 AlphaView 프로젝트 적용 포인트

### 1. **고속 프로토타이핑**
- **현재 문제**: Admin 대시보드 새 차트, SEC 데이터 시각화 같은 UI 실험에 시간 소모
- **Figma Make 활용**: 
  ```
  "S&P 500 인덱스 실시간 차트 + 필터 기능 (날짜, 종목)"
  "내부자 거래 타임라인 (Interactive Timeline)"
  "기관 보유 현황 파이 차트 + 호버 상세 정보"
  ```
  → 1시간 내 인터랙티브 프로토타입 제작 → PM, 팀원 피드백 → 본 개발 착수

### 2. **디자인 시스템 도구 제작**
- **Tools for Making 개념 적용**: 
  - "AlphaView 색상 팔레트 생성기" (브랜드 컬러 기반 변형 생성)
  - "SEC 데이터 카드 레이아웃 제너레이터" (다양한 정보 밀도 테스트)
  - "반응형 그리드 시뮬레이터" (모바일/태블릿/데스크톱 동시 확인)

### 3. **데이터 시각화 실험**
- **미야모토의 "Scrap and Build" 적용**:
  - 5가지 차트 스타일을 각 1분씩 생성 → 즉시 비교 → 최적안 선택
  - 파티클로 주가 변동 표현, 3D로 포트폴리오 비중 표현 같은 **실험적 UI** 빠르게 검증

### 4. **AI와의 협업 문화**
- **칭찬 피드백 도입**: "이 차트 좋네요. 여기에 툴팁 추가해주세요" → 점진적 개선
- **제약 명시**: "이 컴포넌트는 변경하지 말고, 여기만 수정" → 예상치 못한 리그레션 방지

### 5. **실무 적용 로드맵**
```
Phase 1 (1주): Figma Make로 Admin 차트 프로토타입 3종 제작
Phase 2 (1주): 팀 피드백 반영 → 최종안 선정
Phase 3 (2주): React 컴포넌트 구현 (Figma Make 코드 참고)
Phase 4 (1주): 실제 API 연결 + 테스트
```

**기대 효과**: 기존 4주 → **5주** (프로토타입 단계 추가되지만 전체 리스크 감소)
